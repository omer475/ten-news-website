WHAT WE TRIED AND WHY IT DIDN'T FULLY WORK
============================================

Attempt 1: has_more calculation was wrong
  - Old: compared cumulative offset vs shrinking pool → crossed after 5 pages
  - Fix: has_more = selected.length >= limit && remainingPool > selected.length
  - Status: FIXED on server. API returns has_more=true with curl.
  - BUT: user still sees "all caught up" on the simulator app.

Attempt 2: iOS hasMore recovery
  - Added hasMoreBecameFalseAt timestamp, retries after 2 min
  - Status: IMPLEMENTED but doesn't solve the root cause.

Attempt 3: Seen exclusion missing article_exit events
  - 60% of events were article_exit type, not in the exclusion filter
  - Fix: added article_exit, article_revisit, article_liked to the query
  - Status: FIXED. But articles still repeat.

Attempt 4: Pool exhaustion from too many seen IDs
  - User had 335 seen events, pools only had ~200 articles
  - Tried: reducing seen window, widening pools, DB-level exclusion
  - Status: PARTIALLY FIXED. Pools are bigger but still thin for heavy test user.

Attempt 5: Old articles dominating feed (15-19 days old)
  - Personal scoring used recency as additive boost, not multiplier
  - Fix: recency decay now multiplies entire score
  - Also: hard-capped all pools at 7 days
  - Status: FIXED. No articles >7d appear now.

Attempt 6: Seen query LIMIT overflow
  - 366 events in 48h, LIMIT 300 missed events at position 301+
  - Fix: extended to 7d window, LIMIT 1000
  - Status: FIXED on server. curl shows 0 overlap.

Attempt 7: iOS sending stale seen_ids from ReadingHistoryManager
  - ReadingHistoryManager persists 500 entries in UserDefaults across all sessions
  - Initial load sent 100 of these as seen_ids, permanently shrinking pools
  - Fix (professor's instruction): initial load sends NO seen_ids (clean session)
  - Only page 2+ sends within-session seen_ids
  - Status: IMPLEMENTED. API call logs show no seen_ids on initial load.
